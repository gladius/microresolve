# Routing 50 MCP Tools at 60 µs Per Call

**Status:** numbers from a real benchmark — see "How we ran it" at the end for reproducibility.

LLM-based function calling is the default way agents pick which tool to invoke. Send the user's message and the schemas of every available tool to the model; let the model name the right one. It works. It also costs $0.0001–$0.001 per call, takes 200–800 ms, and gets slower as you add tools.

For a class of agents — the ones with stable, well-named tools — you don't need an LLM in the routing path at all. You need an index. This post benchmarks one.

We loaded 50 MCP tools from Smithery (25 GitHub + 25 Slack), gave each one five LLM-generated seed phrases, and sent 40 user-style queries through MicroResolve's lexical router. The headline number:

**100% accuracy at 60 µs per call. $0 per call. Zero LLM at inference.**

The story below is about how we got there, the four bugs the benchmark exposed, and what the architecture buys you that LLM routing doesn't.

---

## Setup

```
Workspace      : mcp-demo (50 intents, 250 seed phrases)
GitHub tools   : create_issue, list_issues, get_issue, close_issue, update_issue,
                 add_issue_comment, create_pull_request, list_pull_requests,
                 merge_pull_request, create_repository, list_repositories,
                 fork_repository, get_file_contents, create_or_update_file,
                 push_files, list_commits, get_commit, create_branch, list_branches,
                 search_repositories, search_code, create_release, list_releases,
                 get_pull_request, delete_file
Slack tools    : slack_send_message, slack_create_channel, slack_list_channels,
                 slack_invite_to_channel, slack_schedule_message, slack_set_status,
                 slack_search_messages, slack_upload_file, slack_remind, ... etc.
Seed phrases   : 5 per tool, generated by Claude Haiku from each tool's description
Test queries   : 40 (20 GitHub + 20 Slack), realistic phrasings
LLM at inference: none
```

The LLM is used twice during setup (once to seed phrases, once to generate test queries) and zero times after. Every routing call is pure index lookup.

## Cold accuracy

The first run, before any continuous learning:

| Domain | Accuracy |
|--------|----------|
| GitHub | 20/20 (100%) |
| Slack  | 16/20 (80%) |
| **Overall** | **36/40 (90%)** |

Latency: median 63 µs, p99 < 200 µs.

90% out of the box, with five seed phrases per tool and zero training. For comparison: an LLM-based router on the same setup hits ~95% but at 200+ ms per call.

The interesting part isn't the 90%. It's the four queries that failed — because each one exposed a real bug in the system.

## The four failures (and what each one taught us)

### 1. "upload the deployment report to slack" → `github:push_files`

The query mentioned Slack explicitly. The router still picked a GitHub tool.

The trace showed why: L1 (the morphology layer) had built a synonym edge `upload → push` during the import. When the LLM seeded `github:push_files`, one of the example phrases was *"upload multiple files at once."* Hebbian learning saw "upload" co-occur with `push_files`, wrote a namespace-level synonym, and now every "upload" query routed to GitHub regardless of context.

The token "slack" in the query barely scored — it appears in every Slack tool's name and description, so its IDF is near zero within the Slack subdomain.

**Fix:** removed the spurious synonym edge from `_l1.json`; removed the misleading seed phrase from `push_files`; added "upload to slack" phrases to `slack_send_message` (since this namespace doesn't include a dedicated file-upload tool). After the fix: routes correctly to `slack_send_message` at score 12.0.

**Generalizable lesson:** Hebbian co-occurrence is a feature, not a bug — it's how the system learns synonyms. But it can over-fit when seed phrases use cross-domain vocabulary. The safety net is continuous learning: the mistake gets caught, fed back, and the wrong edge gets unlearned. We did that fix manually here; in production, the next time a user said "upload to slack" and got the wrong answer, the review queue would surface it.

### 2. "set my status to on vacation" → `github:actions_get`

The trace showed the L0 layer (typo correction) was rewriting the query to "set my status to on **action**" — replacing "vacation" with "action" before any scoring happened.

This is the L0 typo corrector misfiring: "vacation" is 8 chars, "action" is 6 chars, edit distance is 2 (delete `v` and one `a`). That's within the existing edit-distance threshold of 2 for words of length ≥ 5. So L0 treated "vacation" as a typo of "action" and silently substituted.

**Fix:** added a length guard. A typo correction shouldn't shrink a word by more than 1 character — that's not a typo, it's a substring collapse. One line in `src/ngram.rs`:

```rust
if query_chars.len() > term_chars.len() + 1 { return None; }
```

This blocks `vacation→action` and `oncall→call` (the next bug, see below) without rejecting legitimate corrections like `cancell→cancel`.

**Generalizable lesson:** L0 typo correction is one-way, silent, and pre-scoring. Bugs there are invisible in the route trace unless you check `l0_corrected` explicitly. Worth surfacing more prominently in the UI.

### 3. "ping the oncall engineer" → `github:create_branch`

Same L0 bug as #2: `oncall` (6 chars) within edit-distance 2 of `call` (4 chars). L0 stripped `on`, leaving `call`. The token `call` happened to appear in `create_branch`'s training phrase *"call it auth-refactor-v2"*.

The L0 length guard fixed this too. But there was a second issue: the query word `ping` doesn't appear in any Slack tool's vocabulary at all.

**Fix:** added phrases like *"ping the oncall engineer"* and *"ping someone on slack"* to `slack_send_message`. Vocabulary gap closed.

**Generalizable lesson:** Domain-specific jargon (`ping`, `oncall`) needs explicit seeding. The LLM doing initial seed generation tends to produce formal, dictionary phrasings ("send a notification to the on-call team member") and miss the colloquial form ("ping oncall"). Continuous learning from real queries fills this gap; cold-start coverage is fundamentally limited.

### 4. "delete the old messages from the bot" → `github:delete_file`

This one routed correctly in the sense that there *is* a Slack tool called `slack_delete_message` — but the GitHub tool scored 8.39 vs Slack's 1.01. Wrong winner.

The cause was vocabulary imbalance. `github:delete_file` was seeded with five very "delete"-heavy phrases ("delete file", "remove this file", "trash that file", "delete the old script", ...). `slack_delete_message` had six phrases too, but none used "old messages" or "bot" in context. The bigram "delete old" appeared in `delete_file`'s training and not in Slack's.

**Fix:** added contextual phrases to `slack_delete_message` ("delete old messages from the bot", "clean up the bot messages in slack", ...). Now scores 19.05 vs 8.4 — Slack wins decisively.

**Generalizable lesson:** when two intents share vocabulary, the one with denser training wins. The collision-fix feature (`/api/intents/discriminate`) exists for exactly this case and would catch it automatically — given a hint that this pair is colliding. We did it by hand for the benchmark.

## Two infrastructure bugs the benchmark surfaced

The four routing failures above are *intent-design* problems — they show in any routing system. While digging into them, the benchmark exposed two genuine library bugs:

### Bug A — IDF was being recomputed on every query

The original code computed IDF (`ln(N/df)`) per token per query — scanning the inverted index each time. On the 50-intent namespace this took ~330 µs per call out of the 392 µs total. Caching IDF at index-build time and incrementally on phrase-add dropped per-query latency from 392 µs to 63 µs — a 6× speedup that's pure waste removal, not a quality tradeoff.

### Bug B — Phrase removal silently broke the index

`Router::rebuild_l2()` (called whenever a phrase is removed) creates a fresh `IntentIndex` but never called `rebuild_idf()` on it. Result: `intent_count` reset to 0, every IDF computed to `ln(1/n) < 0`, clamped to 0. Every score became 0. Every route returned empty. And the broken state got persisted to disk.

This is the kind of bug you can't see in a unit test that exercises one operation. It shows up when a user removes one phrase and routes get worse for no obvious reason. The fix was one line; finding it required actually running the system.

### Bug C — Newly-created namespaces had `intent_count = 0`

Same root cause as Bug B but a different trigger: `intent_count` was only set by `rebuild_idf()`, which only ran on disk-load. A namespace created in memory (via API calls) never triggered it, so it stayed at 0 until the first server restart. Every route on a fresh namespace returned zero scores.

The fix was to maintain `intent_count` incrementally in `learn_word` using a `known_intents: HashSet<String>` field. Now newly-created namespaces score correctly from the first query.

All three are now covered by unit tests so they can't silently regress.

## What this architecture buys you

The 60 µs / $0 numbers are eye-catching, but the real benefits are structural:

**1. Cost stays flat as you add tools.** LLM-based routing latency and cost grow with the number of tool schemas in context. Lexical routing is O(query tokens), independent of tool count. 50 tools and 5,000 tools route at the same speed.

**2. Latency budget for downstream work.** A 60 µs router leaves 300+ ms of headroom in a typical agent loop for the actual LLM call. A 500 ms router doesn't.

**3. Continuous learning, not retraining.** When the router gets one wrong, you add a phrase. No fine-tune, no eval set, no model deployment. The fix takes effect on the next request.

**4. Inspectable.** Every routing decision has a trace: which tokens matched, which intents scored, what the IDF was. When something goes wrong, you can see why. LLM routers give you a token; this gives you reasoning.

**5. Offline-capable.** Same library compiles to WASM. The router runs in the browser, in a desktop app, on-device — anywhere you don't want to send queries to a cloud API.

---

## What it doesn't buy you

Honest limits:

- **Reasoning over context** — if your tool selection requires understanding nested logical conditions, multi-step planning, or arguments derived from conversation history, you still want the LLM. This router answers "which tool" — not "what arguments" or "in what order."
- **Cold-start without seed phrases** — the router needs vocabulary. Five seed phrases per tool got us 90%; with zero phrases, it routes nothing.
- **Adversarial / out-of-distribution queries** — phrasings that share no vocabulary with any training phrase get no signal. The continuous learning loop catches these as they happen.

The right deployment is **hybrid**: use the lexical router for the 80–95% of queries that match known patterns; fall back to LLM-based routing for the rest. The fallback rate decreases over time as continuous learning adds vocabulary.

## How we ran it

- **Server:** the `microresolve` Rust binary built from this repo (`cargo build --release --bin server --features server`).
- **Workspace:** `mcp-demo`, persisted at `~/.local/share/microresolve/mcp-demo/`.
- **Tools fetched from:** `https://smithery.ai` MCP registry via `/api/import/mcp/fetch?name=github` and `/api/import/mcp/fetch?name=node2flow/slack`.
- **Seeding LLM:** Claude Haiku (`claude-haiku-4-5-20251001`).
- **Benchmark script:** `/tmp/mcp_benchmark.py` (40 queries, threshold 0.1, 1 query at a time).
- **Latency:** measured per-call as `routing_us` field returned by `/api/route_multi`.

To reproduce: see `tests/data/benchmarks/exp_vector_l1.py` for the namespace setup, then run the benchmark script against a local server.
